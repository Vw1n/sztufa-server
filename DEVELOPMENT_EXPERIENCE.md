# 深圳技术大学足球协会系统 (SZTUFA) —— 开发经验与规则总结

本文档总结了从会话开始至今，在开发、调试、提交以及合并过程中积累的宝贵经验与规范规则，用以指导后续的迭代开发。

---

## 1. Git 分支管理与推送规范

### 1.1 Fork 仓库权限隔离
* **背景与经验**：
  在本次协作中，由于对主仓库（`Vw1n/sztu-fa-web.git`）没有直接写入权限，推送代码时遇到了 `403 Permission Denied` 错误。
* **规则**：
  * **必须推送至个人 Fork 仓库**：所有推送命令必须指定 `fork` 远程源（如 `git push fork main`），不可强行推送到 `origin`（即主仓）。
  * **合并流程**：本地开发完后，推送到自己的 Fork 仓，再在 GitHub 网页端点击 **`Contribute -> Open Pull Request`**，向主仓提交合并请求，等待管理员审核合并。

### 1.2 Git 中文 Commit 规范
* **背景与经验**：
  用户要求使用标准的中文 Git 备注描述修改的内容。
* **规则**：
  * 使用规范的 Angular 提交格式：`<type>(<scope>): <subject>`（使用中文描述）。
    * `feat`: 新增功能
    * `fix`: 修复 Bug
    * `refactor`: 重构代码
    * `style`: 样式调整（不影响逻辑）
  * **Windows PowerShell 提交规避 pathspec 错误**：
    在 Windows 环境下，直接在单引号中换行或包含中文字符双引号 `“` 会引发 PowerShell 的解析失败。
    * **避坑写法**：使用多个连续的 `-m` 标志来表达多行内容，且中英文引号要区分。
    * **标准命令示例**：
      ```powershell
      git commit -m "feat: 官网展示端实现赛事回顾卡片及时间轴" `
                 -m "- 修复详情弹窗中比赛时间/地点字体过大的问题" `
                 -m "- 主队事件靠左，客队事件靠右，居中单轴时序排布"
      ```

## 2. CSS 样式污染与隔离规则

* **背景与经验**：
  在 Vite React 项目中，若直接使用非 CSS Modules 方式编写 `.modalHeader`、`.modalOverlay` 等极高重复率的类名，会造成全局样式渗透污染（如 Teams 详情弹窗的样式污染了 Matches 详情弹窗的头部）。
* **规则**：
  * **命名强隔离前缀**：所有组件内的 CSS 命名必须加上该组件的前缀。例如，比赛模块的弹窗一律命名为 `.matchModalOverlay`、`.matchModalHeader`，以绝对避开全局或其他模块的同名冲突。
  * **不要在全局样式中定义具体业务样式**：全局 `index.css` 仅用于放置颜色、字体的 CSS 变量（Variables）及基础 Reset，严禁写入具体卡片或弹窗的布局样式。

## 3. 前端交互与排版精美规则

* **背景与经验**：
  “比赛地点”和“比赛时间”的文字在不同屏幕宽度下极易换行或重叠，原先的字体设计过大显得笨重。
* **排版规则**：
  * **垂直堆叠流**：采用 Emoji + 0.75rem 浅灰色小标签 + 0.9rem 内容文字的**垂直三层堆叠**。避免使用横向 Flex 布局导致文字由于容器宽度不够而被截断或强行换行。
  * **时间轴时序精准计算**：对于足球赛事中带加号的伤停补时（例如 `"45+2'"`），常规 `parseInt` 会解析为 `45`，导致与常规 45 分钟的事件发生顺序错乱。
    * **算法规则**：编写精准的 `parseTime` 函数，解析带 `+` 号的分钟数并加上对应的微小浮点数（例如 `"45+2'"` 转换为 `45.02`），确保在时间轴上做 `sort` 时的排序绝对正确。
  * **双侧时间轴分流**：时间轴指示器在正中央对齐，主队事件靠左，客队事件靠右。使用 Media Queries 编写响应式降级（`max-width: 600px`），使得在小屏（如手机端）时，时间轴自动退化为左侧单边流式排列。

## 4. 后端及 API 架构适配规则

### 4.1 JWT 令牌异步加载防御
* **背景与经验**：
  在 NestJS + TypeORM/Prisma 中，如果 JWT 模块在配置异步读取环境变量时未处理好时序，容易在本地启动时引起签名密钥未初始化，进而导致持续报 `401 Unauthorized` 认证失败错误。
* **规则**：
  * JWT 注册应始终采用 `JwtModule.registerAsync` 结合 `ConfigService` 注入的方式，保障配置项加载完毕后再加载 Auth服务。

### 4.2 接口动态域名自适应
* **背景与经验**：
  为了避免本地开发调试（`localhost`）与线上生产环境切换时手动修改 API 地址的繁琐过程，需要实现服务端的动态切换。
* **规则**：
  * 在管理端 API 封装中（如 `axios` 拦截器），引入自适应规则：
    ```typescript
    const getBaseUrl = () => {
      const hostname = window.location.hostname;
      if (hostname === 'localhost' || hostname === '127.0.0.1') {
        return 'http://localhost:3001/api/v1'; // 本地开发
      }
      return 'https://sztufa-server.yourdomain.com/api/v1'; // 线上生产
    };
    ```

### 4.3 静态资源 WebP 图片直传
* **背景与经验**：
  直接保存 Base64 图片或大图会使数据库体积暴增并拖慢前台渲染。
* **规则**：
  * 前台/后台应配合后端提供流式直传功能（使用 `FormData` 替代 Base64 JSON 串）。
  * 后端采用 `sharp` 对上传的球队 Logo 和球员头像进行 WebP 高损压缩重绘，保障前台页面加载多图时的首屏渲染性能。

## 5. 项目整合开发流程规则

1. **功能审计**：每次接收大型重构需求，必须先通读全局代码、审计 `Prisma Schema`，找出是否存在底层关联约束。
2. **渐进式实施**：严格按照 **后端数据结构重构 -> 后端 API 实现 -> 后台管理页面录入逻辑实现 -> 普通展示端对接呈现** 的生命周期开展开发。
3. **本地合并回归**：分支推送前，必须在本地依次将开发分支（`dev-*`）合入 `main`，并在本地环境中运行 `npm run build` 以验证构建是否有语法或类型错误，防止污染主分支代码。

## 6. 2026年7月9日 权限加固、密码重置与历史赛季名册提取迭代

### 6.1 越权访问防护与 RBAC 控制
* **背景与经验**：
  在本次迭代中，我们发现后台存在只读用户（普通用户）可以越权添加和编辑球队信息的漏洞。原因为后端控制器未加挂权限拦截器，且前端菜单和路由均没有针对 `user` 角色进行有效阻断。
* **规则**：
  * **后端拦截**：所有写操作接口（如 `POST` / `DELETE` / `PATCH`）必须结合 `RolesGuard` 针对具体角色做拦截。
  * **角色专属校验**：例如球队的更新接口，当用户为教练角色时，需要强校验其操作的 `teamId` 必须与用户自身的 `teamId` 一致，防范教练跨队篡改他人球队。
  * **前端守卫**：路由必须使用路由守卫（`RoleGuardRoute`）进行角色限制，且根据用户身份过滤不属于该角色的菜单导航入口。
  * **行内按钮隐藏**：在管理后台列表中，对行内的“编辑”、“删除”等操作按钮进行角色和绑定校验，不满足条件时一律隐藏或禁用。
  * **身份标识容错**：前端在渲染顶部/右上角身份文本时，要避免对非超管/记录员角色做直接的 fallback 降级（即默认归为教练），必须显式精准渲染（超管/记录员/教练/普通用户），防止只读账号展示成教练产生越权错觉。

### 6.2 展示端按赛季回溯球员名册
* **背景与经验**：
  前台展示端要求能查看每支球队在各个赛季的球员名单。但数据库由于结构历史局限，`Player` 表并未关联 `seasonId`，只保留最新活跃状态。
* **规则**：
  * **当前赛季**：直接展示球队当前关联的球员列表（最新名册）。
  * **历史赛季**：对于已归档的历史赛季，通过扫描该队在当赛季的所有比赛事件（包括进球、助攻、得牌、换上换下等事件），提取其中出现的所有球员姓名和球衣号，去重去冗后拼装出历史赛季名册。这在最大程度上保证了老球员毕业销号后，旧赛季战绩中的出场队员依然可追溯查阅。

### 6.3 Prisma 自动合并的多模型重复定义故障
* **背景与经验**：
  在进行分支合并（Merge）时，若使用了自动解决冲突，容易产生 `prisma/schema.prisma` 文件中由于不同的改动而使部分数据模型（如 `Goal`、`MatchEvent`、`News`）被重复生成定义两次。
* **规则**：
  * 每次合并完成后，**必须首先运行 `npx prisma generate` 校验数据模型语法**，定位并删除重复定义的冗余模型，以确保编译正常，防止推送到线上导致 Vercel 部署运行环境编译报 `P1012` 错误进而产生 `FUNCTION_INVOCATION_FAILED` 服务不可用。
  * **多环境驱动一致**：在部署至 Vercel 生产时，`schema.prisma` 里的 `provider` 必须调整为 `postgresql` 并读入 `DATABASE_URL`，不可将本地测试用的 `sqlite` 驱动提交推送，否则引起客户端二进制引擎不匹配导致运行时服务崩溃。
