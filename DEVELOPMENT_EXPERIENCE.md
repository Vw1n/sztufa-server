# 深圳技术大学足球协会系统 (SZTUFA) —— 开发经验与规则总结

本文档总结了从会话开始至今，在开发、调试、提交以及合并过程中积累的宝贵经验与规范规则，用以指导后续的迭代开发。

---

## 1. Git 分支管理与推送规范

### 1.1 Fork 仓库权限隔离
* **背景与经验**：
  在本次协作中，由于对主仓库（`Vw1n/sztu-fa-web.git`）没有直接写入权限，推送代码时遇到了 `403 Permission Denied` 错误。
* **规则**：
  * **必须推送至个人 Fork 仓库**：所有推送命令必须指定 `fork` 远程源（如 `git push fork main`），不可强行推送到 `origin`（即主仓）。
  * **合并流程**：本地开发完后，推送到自己的 Fork 仓，再在 GitHub 网页端点击 **`Contribute -> Open Pull Request`**，向主仓提交合并请求，等待管理员审核合并。

### 1.2 Git 中文 Commit 规范
* **背景与经验**：
  用户要求使用标准的中文 Git 备注描述修改的内容。
* **规则**：
  * 使用规范的 Angular 提交格式：`<type>(<scope>): <subject>`（使用中文描述）。
    * `feat`: 新增功能
    * `fix`: 修复 Bug
    * `refactor`: 重构代码
    * `style`: 样式调整（不影响逻辑）
  * **Windows PowerShell 提交规避 pathspec 错误**：
    在 Windows 环境下，直接在单引号中换行或包含中文字符双引号 `“` 会引发 PowerShell 的解析失败。
    * **避坑写法**：使用多个连续的 `-m` 标志来表达多行内容，且中英文引号要区分。
    * **标准命令示例**：
      ```powershell
      git commit -m "feat: 官网展示端实现赛事回顾卡片及时间轴" `
                 -m "- 修复详情弹窗中比赛时间/地点字体过大的问题" `
                 -m "- 主队事件靠左，客队事件靠右，居中单轴时序排布"
      ```

## 2. CSS 样式污染与隔离规则

* **背景与经验**：
  在 Vite React 项目中，若直接使用非 CSS Modules 方式编写 `.modalHeader`、`.modalOverlay` 等极高重复率的类名，会造成全局样式渗透污染（如 Teams 详情弹窗的样式污染了 Matches 详情弹窗的头部）。
* **规则**：
  * **命名强隔离前缀**：所有组件内的 CSS 命名必须加上该组件的前缀。例如，比赛模块的弹窗一律命名为 `.matchModalOverlay`、`.matchModalHeader`，以绝对避开全局或其他模块的同名冲突。
  * **不要在全局样式中定义具体业务样式**：全局 `index.css` 仅用于放置颜色、字体的 CSS 变量（Variables）及基础 Reset，严禁写入具体卡片或弹窗的布局样式。

## 3. 前端交互与排版精美规则

* **背景与经验**：
  “比赛地点”和“比赛时间”的文字在不同屏幕宽度下极易换行或重叠，原先的字体设计过大显得笨重。
* **排版规则**：
  * **垂直堆叠流**：采用 Emoji + 0.75rem 浅灰色小标签 + 0.9rem 内容文字的**垂直三层堆叠**。避免使用横向 Flex 布局导致文字由于容器宽度不够而被截断或强行换行。
  * **时间轴时序精准计算**：对于足球赛事中带加号的伤停补时（例如 `"45+2'"`），常规 `parseInt` 会解析为 `45`，导致与常规 45 分钟的事件发生顺序错乱。
    * **算法规则**：编写精准的 `parseTime` 函数，解析带 `+` 号的分钟数并加上对应的微小浮点数（例如 `"45+2'"` 转换为 `45.02`），确保在时间轴上做 `sort` 时的排序绝对正确。
  * **双侧时间轴分流**：时间轴指示器在正中央对齐，主队事件靠左，客队事件靠右。使用 Media Queries 编写响应式降级（`max-width: 600px`），使得在小屏（如手机端）时，时间轴自动退化为左侧单边流式排列。

## 4. 后端及 API 架构适配规则

### 4.1 JWT 令牌异步加载防御
* **背景与经验**：
  在 NestJS + TypeORM/Prisma 中，如果 JWT 模块在配置异步读取环境变量时未处理好时序，容易在本地启动时引起签名密钥未初始化，进而导致持续报 `401 Unauthorized` 认证失败错误。
* **规则**：
  * JWT 注册应始终采用 `JwtModule.registerAsync` 结合 `ConfigService` 注入的方式，保障配置项加载完毕后再加载 Auth服务。

### 4.2 接口动态域名自适应
* **背景与经验**：
  为了避免本地开发调试（`localhost`）与线上生产环境切换时手动修改 API 地址的繁琐过程，需要实现服务端的动态切换。
* **规则**：
  * 在管理端 API 封装中（如 `axios` 拦截器），引入自适应规则：
    ```typescript
    const getBaseUrl = () => {
      const hostname = window.location.hostname;
      if (hostname === 'localhost' || hostname === '127.0.0.1') {
        return 'http://localhost:3001/api/v1'; // 本地开发
      }
      return 'https://sztufa-server.yourdomain.com/api/v1'; // 线上生产
    };
    ```

### 4.3 静态资源 WebP 图片直传
* **背景与经验**：
  直接保存 Base64 图片或大图会使数据库体积暴增并拖慢前台渲染。
* **规则**：
  * 前台/后台应配合后端提供流式直传功能（使用 `FormData` 替代 Base64 JSON 串）。
  * 后端采用 `sharp` 对上传的球队 Logo 和球员头像进行 WebP 高损压缩重绘，保障前台页面加载多图时的首屏渲染性能。

## 5. 项目整合开发流程规则

1. **功能审计**：每次接收大型重构需求，必须先通读全局代码、审计 `Prisma Schema`，找出是否存在底层关联约束。
2. **渐进式实施**：严格按照 **后端数据结构重构 -> 后端 API 实现 -> 后台管理页面录入逻辑实现 -> 普通展示端对接呈现** 的生命周期开展开发。
3. **本地合并回归**：分支推送前，必须在本地依次将开发分支（`dev-*`）合入 `main`，并在本地环境中运行 `npm run build` 以验证构建是否有语法或类型错误，防止污染主分支代码。

## 6. 2026年7月9日 权限加固、密码重置与历史赛季名册提取迭代

### 6.1 越权访问防护与 RBAC 控制
* **背景与经验**：
  在本次迭代中，我们发现后台存在只读用户（普通用户）可以越权添加和编辑球队信息的漏洞。原因为后端控制器未加挂权限拦截器，且前端菜单和路由均没有针对 `user` 角色进行有效阻断。
* **规则**：
  * **后端拦截**：所有写操作接口（如 `POST` / `DELETE` / `PATCH`）必须结合 `RolesGuard` 针对具体角色做拦截。
  * **角色专属校验**：例如球队的更新接口，当用户为教练角色时，需要强校验其操作的 `teamId` 必须与用户自身的 `teamId` 一致，防范教练跨队篡改他人球队。
  * **前端守卫**：路由必须使用路由守卫（`RoleGuardRoute`）进行角色限制，且根据用户身份过滤不属于该角色的菜单导航入口。
  * **行内按钮隐藏**：在管理后台列表中，对行内的“编辑”、“删除”等操作按钮进行角色 and 绑定校验，不满足条件时一律隐藏或禁用。
  * **身份标识容错**：前端在渲染顶部/右上角身份文本时，要避免对非超管/记录员角色做直接的 fallback 降级（即默认归为教练），必须显式精准渲染（超管/记录员/教练/普通用户），防止只读账号展示成教练产生越权错觉。

### 6.2 展示端按赛季回溯球员名册
* **背景与经验**：
  前台展示端要求能查看每支球队在各个赛季的球员名单。但数据库由于结构历史局限，`Player` 表并未关联 `seasonId`，只保留最新活跃状态。
* **规则**：
  * **当前赛季**：直接展示球队当前关联的球员列表（最新名册）。
  * **历史赛季**：对于已归档的历史赛季，通过扫描该队在当赛季的所有比赛事件（包括进球、助攻、得牌、换上换下等事件），提取其中出现的所有球员姓名和球衣号，去重去冗后拼装出历史赛季名册。这在最大程度上保证了老球员毕业销号后，旧赛季战绩中的出场队员依然可追溯查阅。

### 6.3 Prisma 自动合并的多模型重复定义故障
* **背景与经验**：
  在进行分支合并（Merge）时，若使用了自动解决冲突，容易产生 `prisma/schema.prisma` 文件中由于不同的改动而使部分数据模型（如 `Goal`、`MatchEvent`、`News`）被重复生成定义两次。
* **规则**：
  * 每次合并完成后，**必须首先运行 `npx prisma generate` 校验数据模型语法**，定位并删除重复定义的冗余模型，以确保编译正常，防止推送到线上导致 Vercel 部署运行环境编译报 `P1012` 错误进而产生 `FUNCTION_INVOCATION_FAILED` 服务不可用。
  * **多环境驱动一致**：在部署至 Vercel 生产时，`schema.prisma` 里的 `provider` 必须调整为 `postgresql` 并读入 `DATABASE_URL`，不可将本地测试用的 `sqlite` 驱动提交推送，否则引起客户端二进制引擎不匹配导致运行时服务崩溃。

## 7. 2026年7月13日 审计日志重构、移动端布局加固与代理网络修复

### 7.1 操作审计日志精构、折叠压缩与智能合并
* **背景与经验**：
  原本系统的审计日志过于模糊，只有宽泛的说明，且包含大量诸如“用户登录”等无数据改动的记录，数据行数庞杂，不利于快速定位具体的关键修改行为。
* **规则与实现**：
  * **删除冗余与无改动日志**：从写入源头与查询层面全面剔除 `USER_LOGIN` 等非数据改动的辅助操作记录，仅在审计日志内保留比赛录入、球员修改、球队管理、用户注册及权限角色变动等核心写操作日志。
  * **极简文本描述**：重构了后台修改描述的输出格式，将复杂的自然语言段落压缩为“操作行为: [对象] -> [变化值]”键值对，例如 `修改球员 "张三" 信息: 姓名: 张三->张小三, 号码: 10->7`，简明易读。
  * **短时连续相同操作合并（聚合展示）**：后端查询审计日志时，会自动将同一个用户在 5 分钟内对同一类型的连续批量操作（例如连续导入多名球员或修改多场比赛）聚合为一条日志显示，如：`批量修改球员 "张三"、"李四" 等 5 人的信息`，大幅精简了页面数据密度。
  * **明细差异动态折叠**：在后台管理页面中，对包含详细属性变动（如比分修改差值、权限变更明细）的日志默认只展示核心简述，附带 `展开修改差异 ▾` 按钮供用户按需点按展开，兼顾了页面紧凑与信息完整度。

### 7.2 后台管理端与权限管理移动端布局自适应优化
* **背景与经验**：
  后台的用户权限管理清单列数较多（包含账号、角色、绑定球队、创建时间及三个操作按钮），在手机窄屏上极易发生换行重叠或按钮挤压。同时新建用户的输入框在手机上没有自适应铺满。
* **规则与实现**：
  * **卡片化表格布局**：在移动端屏幕（`max-width: 768px`）下，使用 CSS 媒体查询将原本横向排列的表格强制转换为垂直卡片块流式展示（`display: block !important`）。
  * **宽屏自适应与下拉框容错**：拉宽所有下拉选择框及操作按钮为 100% 宽度，并优化点击区域，杜绝了移动端上的手指误触。
  * **新建用户表单垂直堆叠**：新建账号表单中的输入控件在移动端降级为单列垂直堆叠，按钮全宽，完美适应手机触屏操作。
  * **长文本描述自动换行对齐**：由于原先的表格单元格含有 `white-space: nowrap` 规则，导致长文本（如修改日志里的长句比对）在移动端卡片中会单行溢出。为此建立了专用的 `.audit-logs-table` 样式，强制在移动端激活 `white-space: normal`，实现长文本的自动换行与优雅排版。

### 7.3 Windows 开发环境下 Localhost IPV6 代理连接拒绝（ECONNREFUSED）故障修复
* **背景与经验**：
  在 Windows 环境下，Node.js 18+ 对 `localhost` 的解析在某些网络协议栈中会优先映射为 IPv6 环回地址 `[::1]`，而当本地 NestJS 仅绑定或监听 IPv4 时，Webpack Dev Server 代理在请求转发时会出现 `ECONNREFUSED` 报错，导致前端接口抓取返回 HTTP 504 错误页面，进而导致 JSON 解析失败抛出 `Unexpected token 'E'... is not valid JSON`。
* **规则与实现**：
  * **显式指定 IPV4 地址**：在所有代理 and 转发配置文件（包括后台管理端 `webpack.config.js` 的 `devServer.proxy` 目标，以及官网展示端 `vite.config.ts` 的 `server.proxy` 目标）中，**必须将 target 显式指定为 `http://127.0.0.1:3001`**，跳过本机的 IPV6 解析过程，保障本地代理连接绝对稳定。

## 8. 2026年7月15日 球队与球员管理界面移动端排版美化、批量保存进度条与比较逻辑加固

### 8.1 球员编辑与录入表格移动端窄屏防挤压排版
* **背景与经验**：
  在手机等移动端（窄屏）设备下，由于表格列没有显式指定列宽限制，导致编辑状态下的 `input` 控件和列名（姓名、学号等）被严重压缩变形，字宽仅剩一个字符宽，且无法横向滚动，导致根本无法进行任何手动修改与录入。
* **规则与实现**：
  * **行内列宽强制约束**：在编辑与展示球员的表格中，为关键的 `<th>` 表头组件显式设置 `width` 与 `minWidth`（例如：姓名设置 `120px`，学号设置 `160px`，球衣设置 `100px` 等），依靠表格本身的结构约束列宽，防止自适应收缩。
  * **表单控件拉伸适配**：为表格行内的所有 `input` 输入框与 `select` 下拉选择控件统一应用 `width: '100%'` 与 `boxSizing: 'border-box'` 属性，使其能够优雅地填充约束后的列宽。
  * **滚动包裹套件**：确保所有大列数表格（如球队列表与录入球员名单）均被 `<div className="player-table-wrapper">` 包裹，激活 CSS 中的 `.player-table-wrapper { overflow-x: auto; }`，使其可以在窄屏下进行水平滑动操作，从而彻底解决被挤压变形的问题。

### 8.2 球员批量导入与保存的可视化进度条浮层
* **背景与经验**：
  在录入或修改球队信息时，通过 Excel 导入球员后，保存时前端会循环调用 API 发起网络请求。如果球员数量过多，同步保存阶段需要耗时数秒，此时若缺乏直观的进度指示，用户可能会以为系统卡顿而频繁点击保存，导致数据录入冲突。
* **规则与实现**：
  * **分步同步状态更新**：在 `TeamInfoPage.tsx` 和 `MatchSchedulePage.tsx` 中定义 `saveProgress` 状态。在执行多步骤循环（如：创建球队 -> 循环新增球员 -> 循环更新球员）时，实时更新已完成的步数、总步数与当前正在操作的具体球员数据信息。
  * **高颜值毛玻璃磨砂浮层**：在 React 的渲染树底部，插入一个全屏浮层遮罩 `progress-overlay`，采用毛玻璃滤镜（`backdropFilter: 'blur(4px)'`），中间承载进度卡片展示同步球员详情及动态百分比进度条。遮罩层具有全局层级（`zIndex: 1000`），同步期间能够有效拦截用户的其他一切点击误操作。

### 8.3 球员差异比对逻辑中的 NaN 比较漏洞修复
* **背景与经验**：
  在进行数据更新同步前，前端需要比对哪些球员的信息发生了更改（决定是否调用 `playerApi.update`）。原比对逻辑采用了 `Number(original.yellowCards) !== Number(p.yellowCards)` 类似的做法。但如果一个球员在数据库和编辑数据中都没有卡片（都为 `undefined` 或 `null`），`Number(undefined)` 返回 `NaN`，而在 JS 中，`NaN !== NaN` 的判断恒为 `true`，从而导致无更新的球员也会被误判为“数据有改动”，产生了多余的批量保存 API 请求。
* **规则与实现**：
  * **默认值空安全转换**：比对数值前，必须对可能缺失的字段使用 `|| 0` 转换为基线零值，如 `Number(original.yellowCards || 0) !== Number(p.yellowCards || 0)`；对状态字段使用 `|| 'active'` 等进行自适应容错判断，确保比对绝对精准，保护后端接口不被多余的无用更新请求拖累。
